# Appendix A — Glossary

Formal terms defined in the architecture design documents. Each term is capitalized on first use in each document. Terms are listed alphabetically with cross-references to the section where they are defined.

---

| Term | Definition | Defined In |
|------|-----------|-----------|
| **Abstract Machine** | The complete computational model: Machine Level Registry + Execution Layers + Functions + Tasks + Communication. | [02 §2.1.1](02-logical-view/01-domain-model.md#211-the-abstract-machine) |
| **Address Space** | Per-Memory-Scope set of local addresses. A Global Address is `(level_path[], local_address)`. | [02 §2.1.5](02-logical-view/04-memory.md#215-memory-scope) |
| **Admission (Submission)** | Atomic acceptance of a Submission by a Scheduler. Either every Task slot + every intra-group edge + the Group Workspace (if any) is allocated, or the Submission is rejected as a whole. Gated by the Outstanding Submission Window. | [02 §2.4.A](02-logical-view/07-task-model.md#24a-submission-model), [02 §2.1.3.1.A](02-logical-view/02-scheduler.md#2131a-submission-admission--outstanding-window) |
| **AICore Function** | A Function executing on one or more AICores at a leaf Machine Level. Does not submit child Tasks. | [02 §2.3.1](02-logical-view/06-function-types.md#231-aicore-function) |
| **Boundary Task** | A Task inside a Group Submission whose inputs are produced outside the Submission (`boundary_in`) or whose outputs are consumed outside the Submission (`boundary_out`). External dependency edges and boundary-input/output tensors are attached only to boundary Tasks. | [02 §2.4.A](02-logical-view/07-task-model.md#24a-submission-model), [02 §2.4.C](02-logical-view/07-task-model.md#24c-group-submission-semantics) |
| **Channel** | Typed communication path between two endpoints. Either Vertical (parent↔child) or Horizontal (sibling↔sibling). | [02 §2.5](02-logical-view/08-communication.md#25-communication-model) |
| **Child Layer** | Layer `Lₖ₋₁` relative to its Parent Layer `Lₖ`. | [02 §2.1.2](02-logical-view/01-domain-model.md#212-execution-layer) |
| **ContinuousTensor** | Tensor descriptor with `shapes` (storage layout) and `valid_shapes` (logical extent). | [02 §2.4.3](02-logical-view/07-task-model.md#243-task-arguments-and-continuoustensor) |
| **Core Wrap** | Canonical Worker Group on Ascend NPU: 1 AIC (cube core) + 2 AIV (vector cores), non-overlapping. Tasks requiring multiple core types must use cores within the same Core Wrap. | [02 §2.1.4.2](02-logical-view/03-worker.md#2142-heterogeneous-worker-types-and-worker-groups) |
| **Data Access Model** | Shared memory or distributed message-passing patterns for cross-scope data access. | [02 §2.5.5](02-logical-view/08-communication.md#255-data-access-models) |

| **Dedicated Deployment Mode** | Scheduler–Worker deployment where separate physical execution units handle scheduling and task execution. Default mode. | [02 §2.1.3.6](02-logical-view/02-scheduler.md#2136-schedulerworker-deployment-flexibility) |
| **Deferred Event Handling** | Event handling mode where a `SchedulerEvent` is queued internally for later batch processing rather than handled inline. | [02 §2.1.3.5](02-logical-view/02-scheduler.md#2135-event-driven-execution-model) |
| **Dependency Mode** | Per-Submission attribute (`BARRIER`, `DATA`, or `NONE`) that controls how the Scheduler attaches external dependency edges from this Submission's boundary-in Tasks to Tasks in previously admitted, not-yet-retired Submissions. Intra-Submission edges are always installed regardless of mode. `DATA` mode additionally requires the per-Layer dep-metadata lock during admission because it reads and writes `producer_index` and fan-in counters concurrently with prior-task completion handlers; `BARRIER` and `NONE` modes require no such lock. | [02 §2.4.B](02-logical-view/07-task-model.md#24b-dependency-modes), [02 §2.1.3.1.B](02-logical-view/02-scheduler.md#2131b-dependency-resolution-per-submission) |
| **Dependency Model** | Specification module defining the split contract between the frontend dependency analyzer (intra-Submission RAW / WAR / WAW / assemble overlap, lowered into `intra_edges`) and the runtime `TaskManager` (cross-Submission RAW via `producer_index`). Not a code module — enforced by frontend + `scheduler/` + `memory/`. v1 scope: runtime RAW-only. | [02 §2.10](02-logical-view/12-dependency-model.md), [08 ADR-013](08-design-decisions.md#adr-013-two-stage-dependency-analysis--frontend-intra-submission-dag--runtime-cross-submission-raw) |
| **Deployment** | The process of instantiating the Abstract Machine for a specific hardware setup. | [05 §5.5](05-physical-view.md#55-deployment-configuration) |
| **Distributed Orchestration Function** | A Function at a distributed Machine Level that coordinates work across sibling instances. | [02 §2.3.4](02-logical-view/06-function-types.md#234-distributed-orchestration-function) |
| **Distributed Scheduler** | ISchedulerLayer implementation for distributed levels; partitions and routes Tasks across nodes. | [02 §2.6.2](02-logical-view/09-interfaces.md#262-distributed-scheduler) |
| **DSL Level Alias (`pl.Level`)** | Configurable mapping from user-facing DSL enum to Machine Level name. | [02 §2.2.3](02-logical-view/05-machine-level-registry.md#223-dsl-level-alias-pllevel) |

| **Event Loop** | The Scheduler's sole execution entry point: a loop that collects events from registered sources, dispatches handlers, processes deferred events, and consults `IExecutionPolicy`. | [02 §2.1.3.5](02-logical-view/02-scheduler.md#2135-event-driven-execution-model), [04 §4.1.4](04-process-view.md#414-scheduler-event-loop) |
| **Execution Layer** | Recursive building block: `(Scheduler, Workers[], MemoryScope)`. | [02 §2.1.2](02-logical-view/01-domain-model.md#212-execution-layer) |

The runtime does not perform intra-Submission hazard detection. | [02 §2.10.2](02-logical-view/12-dependency-model.md#2102-frontend-contract), [08 ADR-013](08-design-decisions.md#adr-013-two-stage-dependency-analysis--frontend-intra-submission-dag--runtime-cross-submission-raw) |
